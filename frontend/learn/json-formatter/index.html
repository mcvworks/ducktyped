JSON Explained

Understand JSON syntax rules, common validation pitfalls, the difference between formatting and minifying, and best practices for working with JSON data in APIs and configuration files.

What Is JSON?

JSON (JavaScript Object Notation) is a lightweight, text-based data interchange format. Despite its name, JSON is completely language-independent and is supported natively or through libraries in virtually every programming language in use today.

JSON is both human-readable and machine-parseable, which makes it an excellent choice for transmitting structured data between systems. It is formally defined in two standards: RFC 8259 (by the IETF) and ECMA-404 (by Ecma International).

Originally derived from a subset of JavaScript object literal syntax, JSON rose to dominance in the mid-2000s as web APIs shifted away from XML. Today, it is the standard data format for REST APIs, configuration files, NoSQL databases, and inter-service communication. Its simplicity and universal support make it the default choice for data exchange on the modern web.

JSON Syntax Rules

JSON has a small, strict set of syntax rules. Unlike JavaScript, JSON does not tolerate shortcuts or informal syntax. Every valid JSON document must follow these rules exactly:

  • Data is organized in key/value pairs.
  • Keys must be double-quoted strings. Single quotes and unquoted keys are not allowed.
  • Values can be one of six types: string, number, boolean (true/false), null, object, or array.
  • Objects are wrapped in curly braces {}.
  • Arrays are wrapped in square brackets [].
  • Items are separated by commas -- but trailing commas are not allowed.
  • No comments of any kind. Unlike JavaScript, JSON does not support // or /* */ comments.
  • Strings must use double quotes only. Single quotes are invalid.
  • Values like undefined, NaN, Infinity, and functions are not valid in JSON.

Here is a valid JSON example demonstrating several value types:

{
  "name": "Alice",
  "age": 32,
  "isActive": true,
  "address": {
    "city": "Portland",
    "zip": "97201"
  },
  "tags": ["developer", "admin"],
  "deletedAt": null
}

Common JSON Errors

Most JSON validation failures come from a handful of recurring mistakes. These are the errors you will encounter most often:

  • Trailing commas after the last item in an object or array. This is valid in JavaScript but not in JSON: 
    {"a": 1, "b": 2,}
  • Single quotes instead of double quotes: 
    {'key': 'value'}
  • Unquoted keys -- this is JavaScript object syntax, not JSON: 
    {key: "value"}
  • Comments of any kind: 
    // this is not allowed
/* neither is this */
  • Missing quotes on string values -- bare words are not valid unless they are true, false, or null.
  • Using undefined or NaN -- neither exists in the JSON specification.

JSON Data Types

JSON supports exactly six data types. Understanding the rules for each one helps you write valid JSON and debug problems faster.

Strings

Strings are enclosed in double quotes and support standard escape sequences: \n (newline), \t (tab), \" (literal quote), \\ (backslash), and \uXXXX (Unicode code point). Raw control characters are not allowed inside strings.

Numbers

Numbers can be integers or floating-point values. Leading zeros are not permitted (e.g., 007 is invalid). Scientific notation is supported (1.5e10), but hexadecimal (0xFF) and octal (0o77) literals are not. NaN and Infinity are also excluded from the spec.

Booleans

Boolean values are true and false, always lowercase. True, False, TRUE, and FALSE are all invalid. This is a common source of errors when converting from Python, which uses capitalized True/False.

Null

The null type is represented by the literal null, always lowercase. None (Python), NULL (SQL), and nil (Ruby/Go) are not valid JSON.

Objects

Objects are unordered collections of key/value pairs enclosed in curly braces. While many implementations preserve insertion order, the JSON specification does not guarantee it. Keys must be unique strings, and duplicate keys produce undefined behavior across parsers.

Arrays

Arrays are ordered lists of values enclosed in square brackets. Unlike objects, element order is guaranteed. Arrays can contain mixed types -- strings, numbers, objects, other arrays, and nulls can all appear in the same array.

Formatting vs Minifying

JSON whitespace is insignificant to parsers, which means the same data can be represented in a compact or expanded form. The two operations are:

Formatting (Pretty-Printing)

Formatting adds indentation and line breaks to make JSON human-readable. The most common indentation levels are 2 spaces or 4 spaces. Formatted JSON is easier to scan, debug, and compare in diffs. It is the preferred form for configuration files, documentation, and development environments.

Minifying

Minifying strips all unnecessary whitespace -- spaces, tabs, and newlines -- from a JSON document. The result is a single, dense line of text. This reduces file size, which matters for network transfer and storage. A 10KB formatted JSON file might shrink to around 6KB when minified. Production APIs and log pipelines typically use minified JSON.

JSON in Practice

JSON appears in nearly every layer of modern software development:

  • REST APIs -- JSON is the default response format for web APIs. Responses are served with the Content-Type: application/json header.
  • Configuration files -- Widely used in tooling: package.json (Node.js), tsconfig.json (TypeScript), .eslintrc.json (ESLint), composer.json (PHP).
  • Data storage -- NoSQL databases like MongoDB store documents in BSON (Binary JSON), a binary-encoded superset of JSON. PostgreSQL and MySQL also support native JSON column types.
  • Web storage -- Browser APIs like localStorage and sessionStorage store string values, so developers use JSON.stringify() and JSON.parse() to persist structured data.
  • Data exchange -- JSON is the common language between microservices, between frontend and backend, and between third-party integrations. Its simplicity makes it easy to produce and consume across language boundaries.

JSON vs Similar Formats

JSON is not the only structured data format. Here is how it compares to its most common alternatives:

  • YAML -- A superset of JSON that allows comments, multiline strings, and indentation-based nesting. More human-friendly for writing by hand. Widely used in CI/CD configs (GitHub Actions, Docker Compose, Kubernetes manifests). The flexibility comes at the cost of more complex parsing rules.
  • XML -- More verbose than JSON, with support for attributes, namespaces, schemas, and processing instructions. Still dominant in enterprise environments, SOAP APIs, and document formats (XHTML, SVG, RSS). Higher overhead per byte of actual data.
  • TOML -- Simpler and more opinionated than YAML, designed specifically for configuration files. Used in Rust (Cargo.toml), Python (pyproject.toml), and Hugo. Supports comments and is less ambiguous than YAML, but less widely supported for data interchange.
