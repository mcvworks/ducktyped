URL Encoding Explained

Understand what percent-encoding is, why URLs require it, how reserved and unreserved characters differ, and how to correctly encode query parameters, path segments, and international characters in your web applications.

What Is URL Encoding?

URL encoding -- also called percent-encoding -- is a mechanism for representing characters in a URI (Uniform Resource Identifier) that would otherwise be unsafe, ambiguous, or invalid. The formal specification is defined in RFC 3986, which governs the syntax of URIs on the web.

URIs are restricted to a limited set of ASCII characters. When a URL needs to contain characters outside this set -- such as spaces, special symbols, or non-ASCII Unicode characters -- those characters must be replaced with a percent sign (%) followed by two hexadecimal digits representing the character's byte value. For example, a space becomes %20, and a forward slash becomes %2F.

Without URL encoding, a browser or server would have no reliable way to distinguish between characters that are part of the URL's structure (like ? and &) and characters that are part of the actual data being transmitted. Percent-encoding resolves this ambiguity.

Reserved vs Unreserved Characters

RFC 3986 divides ASCII characters into two categories that determine whether they need encoding:

Unreserved Characters

These characters can appear in a URI without encoding. They carry no special structural meaning:

A-Z  a-z  0-9  -  _  .  ~

These 66 characters are safe to use anywhere in a URL. Encoding them is allowed but unnecessary and discouraged, since it increases URL length without adding meaning.

Reserved Characters

These characters have special meaning within the URI syntax and serve as delimiters between components:

:  /  ?  #  [  ]  @  !  $  &  '  (  )  *  +  ,  ;  =

When these characters appear in a URI component as data rather than as delimiters, they must be percent-encoded. For instance, the & character separates query parameters, so if a parameter value itself contains an ampersand, it must be encoded as %26 to prevent the parser from treating it as a separator.

How Percent-Encoding Works

The encoding process is straightforward. Each byte of the character to be encoded is represented as a percent sign followed by two uppercase hexadecimal digits:

Character:   space     &       =       /       @
ASCII code:  32        38      61      47      64
Hex value:   20        26      3D      2F      40
Encoded:     %20       %26     %3D     %2F     %40

The percent sign itself must be encoded as %25 when it appears literally in data. This is essential -- if a URL contains the text 100%, it must be written as 100%25 to avoid the parser interpreting the % as the start of an encoded sequence.

Decoding reverses the process: the parser encounters a %, reads the next two hexadecimal digits, and replaces the triplet with the corresponding byte value.

Encoding Spaces: + vs %20

Spaces in URLs are a common source of confusion because there are two valid ways to encode them, depending on context:

  • %20 -- The standard percent-encoding for a space, valid in all URI components (path, query, fragment). This is the encoding defined by RFC 3986.
  • + -- An alternative encoding for spaces that is only valid in the query string component. This convention comes from the application/x-www-form-urlencoded content type, which is used when HTML forms submit data via GET or POST.

In the query string ?name=John+Doe, the + represents a space. But in a path segment like /users/John+Doe, the + is interpreted as a literal plus sign, not a space. If you mean a space in a path, you must use /users/John%20Doe.

The safest general rule: use %20 for spaces unless you are specifically building form-encoded query strings, in which case + is conventional.

UTF-8 and International Characters

Modern URLs often contain characters beyond the ASCII range -- accented letters, CJK characters, Arabic script, emoji, and other Unicode code points. RFC 3986 mandates that these characters be encoded using their UTF-8 byte representation, with each byte individually percent-encoded.

For example, the character e with an accent (U+00E9) is encoded in UTF-8 as the two bytes 0xC3 0xA9, which becomes %C3%A9 in the URL. A single emoji like a smiley face might produce four bytes and therefore four percent-encoded triplets.

Character:  e (with accent)   (Chinese character for "big")
UTF-8 hex:  C3 A9              E5 A4 A7
Encoded:    %C3%A9             %E5%A4%A7

Browsers display these encoded sequences as their original characters in the address bar -- a convenience feature called Internationalized Resource Identifiers (IRIs). Behind the scenes, the encoded form is what is actually sent in the HTTP request.

encodeURIComponent vs encodeURI in JavaScript

JavaScript provides two built-in functions for URL encoding, and using the wrong one is a frequent source of bugs:

encodeURIComponent()

Encodes a single URI component -- a query parameter value, a path segment, or a fragment identifier. It encodes all characters that are not unreserved, including reserved delimiters like /, ?, &, and =.

encodeURIComponent("price=10¤cy=USD")
// "price%3D10%26currency%3DUSD"

Use this function when you are encoding a value to embed inside a URL. It is the correct choice in the vast majority of cases.

encodeURI()

Encodes a complete URI. It does not encode reserved characters that are structurally meaningful in a full URL, such as :, /, ?, #, and &.

encodeURI("https://example.com/path?name=John Doe")
// "https://example.com/path?name=John%20Doe"

Use this function only when you have a complete URL that already has its structural delimiters in place and you simply want to encode any spaces or non-ASCII characters within it. Using encodeURI() on individual parameter values will leave dangerous characters unescaped.

Common Encoding Scenarios

Understanding when and where to apply URL encoding prevents many common web development bugs:

  • Query parameters -- Each parameter key and value should be individually encoded with encodeURIComponent(), then joined with = and &: 
    const key = encodeURIComponent("search query");
const val = encodeURIComponent("cats & dogs");
const url = `/api?${key}=${val}`;
// "/api?search%20query=cats%20%26%20dogs"
  • Path segments -- User-provided values in URL paths must be encoded to prevent slashes and special characters from breaking the routing structure. A username like user/admin must become user%2Fadmin.
  • Form data -- When an HTML form submits with method="GET", the browser automatically encodes the form fields using application/x-www-form-urlencoded format, which uses + for spaces.
  • Hash fragments -- The fragment portion of a URL (after #) is never sent to the server, but it should still be encoded for the browser to parse it correctly.

The Double Encoding Pitfall

Double encoding occurs when data that is already percent-encoded gets encoded a second time. The % sign in the already-encoded string is itself encoded to %25, producing mangled results:

Original:       hello world
First encoding: hello%20world
Double encoded: hello%2520world  (the % became %25)

This typically happens when encoding is applied at multiple layers -- for example, manually encoding a value before passing it to a library that also encodes it, or when a framework's routing layer double-encodes path parameters.

To avoid double encoding: encode data exactly once, as close to the point of URL construction as possible. If you are unsure whether a string is already encoded, try decoding it first -- if decoding changes the string, it was encoded; if not, it was raw.

application/x-www-form-urlencoded

This content type is the default encoding for HTML form submissions. It is closely related to standard percent-encoding but has a few differences:

  • Spaces are encoded as + instead of %20.
  • The encoding follows the rules of the HTML specification rather than RFC 3986.
  • Key/value pairs are separated by &, and keys are separated from values by =.

When a form with method="POST" and no explicit enctype is submitted, the browser sends the form data in the request body using this format. The result looks like: username=John+Doe&password=s%26cret.

In JavaScript, the URLSearchParams API handles this format natively and takes care of encoding and decoding automatically.
