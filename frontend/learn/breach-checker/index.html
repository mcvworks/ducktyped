Password Breach Checking Explained

Understand how data breaches expose credentials, how the Have I Been Pwned service and k-anonymity model let you check passwords without revealing them, and what steps to take if your accounts have been compromised.

What Is a Data Breach?

A data breach occurs when an unauthorized party gains access to a system and extracts sensitive data -- typically user credentials, personal information, or financial records. Breaches can result from exploited software vulnerabilities, misconfigured databases, phishing attacks against employees, or insider threats. Once data is exfiltrated, it is typically sold on underground markets, shared on forums, or published openly on paste sites.

Data breaches are not rare events. They happen at a staggering scale. Major incidents have exposed hundreds of millions of records at a time: the Yahoo breaches (2013-2014) compromised 3 billion accounts, the LinkedIn breach (2012) exposed 164 million credentials, and the Collection #1 dump (2019) aggregated 773 million unique email addresses from multiple sources. Smaller breaches occur constantly across forums, startups, gaming platforms, and e-commerce sites, often without the affected company even realizing it.

The cumulative result is that billions of username-password pairs are circulating in breach databases. If you have used the internet for any length of time, there is a meaningful probability that at least one of your credentials has been exposed. Checking whether your passwords appear in known breaches is one of the most impactful security steps you can take.

Have I Been Pwned (HIBP)

Have I Been Pwned is a free service created by security researcher Troy Hunt in 2013. It aggregates data from known breaches and provides a searchable database that lets individuals check whether their email addresses or passwords have been exposed. As of 2024, HIBP contains data from over 700 breached sites and more than 12 billion compromised accounts.

HIBP offers two primary lookup services:

  • Email/account search -- checks whether a given email address appears in any known breach dataset. This tells you which services were breached and what categories of data were exposed (passwords, IP addresses, physical addresses, etc.).
  • Password search (Pwned Passwords) -- checks whether a specific password has ever appeared in any breach, regardless of which account it was associated with. This is the service that powers duckTyped's Password Breach Checker.

The Pwned Passwords database contains over 800 million unique passwords collected from breach dumps. If a password appears in this list, it means attackers already have it in their dictionaries and will try it in credential stuffing and brute-force attacks.

How k-Anonymity Works

The obvious concern with checking a password against a breach database is: does the check itself expose the password? If you send your password to a server for comparison, you have just shared your secret with a third party. The Pwned Passwords API solves this problem elegantly using a technique called k-anonymity.

Here is exactly how the process works:

  1. Hash the password locally -- your browser computes the SHA-1 hash of the password on your device. For example, the password P@ssw0rd produces the hash 21BD12DC183F740EE76F27B78EB39C8AD972A757. SHA-1 is a one-way function -- you cannot reverse the hash to recover the password.
  2. Send only the first 5 characters -- instead of sending the full 40-character hash, only the first 5 characters (the hash prefix) are sent to the API. In this example, only 21BD1 is transmitted.
  3. Receive a list of matching suffixes -- the API returns all hash suffixes in its database that share the same 5-character prefix, along with breach counts. This list typically contains 500 to 800 entries.
  4. Compare locally -- your browser searches the returned list for the remaining 35 characters of your hash. If a match is found, the password has appeared in a breach. If not, it has not been seen in any known breach.

The critical insight is that the API server never learns your actual password or even its full hash. It sees only a 5-character prefix that matches hundreds of other passwords. This is the k-anonymity property: your query is indistinguishable from at least k other possible queries, making it impossible for the server to determine which password you were checking.

Why This Is Privacy-Safe

The k-anonymity model provides strong privacy guarantees for several reasons:

  • No plaintext transmission -- the password itself is never sent over the network. Only a 5-character hash prefix is transmitted, and SHA-1 is irreversible.
  • Ambiguity by design -- the 5-character prefix matches hundreds of different passwords in the database. The server cannot distinguish your password from any of the others.
  • All comparison is local -- the final determination of whether your password has been breached happens entirely in your browser. The result is never communicated back to the server.
  • No logging risk -- even if the API server logs every request, the logs contain only hash prefixes, which are useless for recovering passwords.

This approach has been endorsed by organizations including NIST, which recommends checking passwords against known breach lists as part of its digital identity guidelines (NIST SP 800-63B). The k-anonymity model makes this check practical without introducing new privacy risks.

What to Do If Your Password Is Breached

If the breach checker indicates that your password has been found in a breach database, you should take immediate action:

  1. Change the password immediately -- on every account where you used that password. Do not reuse the breached password anywhere. Replace it with a long, randomly generated password created by a password manager or a tool like duckTyped's Password Generator.
  2. Enable two-factor authentication (2FA) -- wherever it is available. Even if an attacker has your password, 2FA requires a second verification step (typically a time-based code from an authenticator app) that the attacker does not have. Prefer authenticator apps (like Google Authenticator or Authy) over SMS-based 2FA, which is vulnerable to SIM-swapping attacks.
  3. Check other accounts -- if you reused the breached password on multiple services, every one of those accounts is at risk. This is the single biggest danger of password reuse and the primary reason security experts recommend unique passwords for every account.
  4. Review account activity -- check for unauthorized logins, unfamiliar devices, or unexpected changes to account settings (email address, phone number, recovery options). Many services provide login history in their security settings.
  5. Consider a password manager -- tools like 1Password, Bitwarden, or KeePass generate and store unique, strong passwords for every account. You only need to remember one master password.

Credential Stuffing Attacks

Credential stuffing is an automated attack where adversaries take username-password pairs from one breach and systematically try them against other services. The attack exploits a simple reality: most people reuse passwords across multiple accounts. If your LinkedIn password was exposed in a breach and you used the same password for your email, an attacker can gain access to your email account without ever specifically targeting you.

Credential stuffing is carried out at massive scale using botnets and specialized tools. Attackers test millions of credential pairs per hour across thousands of websites. Success rates are typically between 0.1% and 2%, but given the volume of attempts, this translates to millions of compromised accounts. The breached password databases that power these attacks are the same ones checked by tools like HIBP's Pwned Passwords.

The most effective defense against credential stuffing is simple: never reuse passwords. If every account has a unique password, a breach at one service cannot cascade to others.

Breach Notification Services

Beyond one-time password checks, several services provide ongoing monitoring and alerts:

  • Have I Been Pwned notification -- you can register your email address with HIBP to receive automatic alerts when it appears in a newly discovered breach.
  • Browser-integrated checks -- Google Chrome, Firefox, and Safari all include built-in password monitoring that warns you if saved credentials appear in known breaches.
  • Password manager audits -- most password managers include a security audit feature (sometimes called "Watchtower" or "Security Dashboard") that checks all stored passwords against breach databases and flags weak, reused, or compromised credentials.
  • Enterprise monitoring -- organizations can use domain-level monitoring to detect when employee credentials appear in breaches, enabling proactive password resets before attackers exploit the data.

Proactive monitoring is significantly more effective than periodic manual checks because breaches are discovered continuously, and the window between a breach and exploitation by attackers can be very short.
