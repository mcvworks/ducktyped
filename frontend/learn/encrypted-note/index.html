Encrypted Notes Explained

Understand how end-to-end encryption protects sensitive messages, how AES-GCM and PBKDF2 work together to secure your data, and how duckTyped's Secure Note Sender lets you share secrets without exposing plaintext to any server.

What Is End-to-End Encryption?

End-to-end encryption (E2EE) is a method of securing data so that only the sender and the intended recipient can read it. When a message is end-to-end encrypted, it is encrypted on the sender's device before it leaves and can only be decrypted on the recipient's device. No intermediary -- not the server hosting the data, not the network provider, not even the service operator -- can access the plaintext content.

This stands in contrast to encryption in transit (like HTTPS), which protects data while it travels between your device and a server but leaves the server able to read the content once it arrives. With E2EE, the server only ever handles ciphertext -- encrypted data that is computationally infeasible to decrypt without the correct key.

End-to-end encryption is used in messaging apps like Signal, in file-sharing services, and in tools like duckTyped's Secure Note Sender. The core principle is always the same: the encryption key never touches the server.

How the Secure Note Tool Works

The duckTyped Secure Note Sender implements a zero-knowledge architecture where all encryption and decryption happens entirely in your browser. Here is the process step by step:

  1. You write your message in the input field. The plaintext never leaves your browser.
  2. A cryptographic key is derived from a randomly generated passphrase or password using PBKDF2 (Password-Based Key Derivation Function 2).
  3. The message is encrypted using AES-GCM (Advanced Encryption Standard in Galois/Counter Mode) with the derived key. A random initialization vector (IV) is generated for each encryption operation.
  4. The ciphertext is sent to the server for storage. The server receives only the encrypted blob -- it has no access to the key or the plaintext.
  5. A shareable link is generated that contains the decryption key in the URL fragment (the part after the # symbol). URL fragments are never sent to the server in HTTP requests, so the key remains client-side only.
  6. The recipient opens the link, and their browser extracts the key from the URL fragment, fetches the ciphertext from the server, and decrypts the message locally.

At no point does the server have access to both the ciphertext and the key simultaneously. This is the foundation of the tool's security model.

Symmetric vs. Asymmetric Encryption

Encryption algorithms fall into two broad categories, and understanding the distinction is important for evaluating any encryption tool.

Symmetric encryption uses a single key for both encryption and decryption. The sender and recipient must both possess the same secret key. AES is the most widely used symmetric cipher. Symmetric encryption is fast and efficient, making it ideal for encrypting large amounts of data. The challenge is key distribution -- how do you securely share the key with the recipient?

Asymmetric encryption (also called public-key cryptography) uses a pair of mathematically related keys: a public key for encryption and a private key for decryption. RSA and elliptic-curve cryptography (ECC) are common asymmetric algorithms. Asymmetric encryption solves the key distribution problem because the public key can be shared openly, but it is significantly slower than symmetric encryption.

In practice, most secure systems use a hybrid approach: asymmetric encryption to exchange a symmetric key, then symmetric encryption for the actual data. The Secure Note Sender uses a simpler model -- pure symmetric encryption with the key embedded in the URL fragment -- because the link itself serves as the secure key distribution channel.

AES-GCM Explained

AES (Advanced Encryption Standard) is a symmetric block cipher adopted by the U.S. National Institute of Standards and Technology (NIST) in 2001. It operates on fixed-size 128-bit blocks and supports key sizes of 128, 192, or 256 bits. AES has withstood over two decades of cryptanalysis and remains the gold standard for symmetric encryption.

GCM (Galois/Counter Mode) is an authenticated encryption mode that provides both confidentiality and integrity. This means AES-GCM not only encrypts your data but also produces an authentication tag that detects any tampering with the ciphertext. If an attacker modifies even a single bit of the encrypted data, decryption will fail rather than producing corrupted plaintext.

AES-GCM requires three inputs:

  • Key -- the symmetric encryption key (typically 256 bits for maximum security).
  • Initialization Vector (IV) -- a random 96-bit value that ensures the same plaintext encrypted with the same key produces different ciphertext each time. The IV must never be reused with the same key.
  • Plaintext -- the data to encrypt.

The output is the ciphertext plus a 128-bit authentication tag. AES-GCM is the cipher suite recommended by TLS 1.3, and it is natively supported by the Web Crypto API in all modern browsers, which is what makes client-side encryption tools like this one possible.

Key Derivation with PBKDF2

Raw passwords or passphrases are not suitable as encryption keys directly. They tend to have low entropy (predictable patterns, limited character sets) and are not the right length for AES. PBKDF2 (Password-Based Key Derivation Function 2) solves this by transforming a password into a cryptographically strong key.

PBKDF2 works by applying a pseudorandom function (typically HMAC-SHA-256) to the password along with a salt -- a random value unique to each encryption operation -- and repeating this process for a specified number of iterations. The output is a key of the desired length.

  • Salt -- prevents precomputed attacks (rainbow tables). Even if two users choose the same password, different salts produce different keys. The salt does not need to be secret and is stored alongside the ciphertext.
  • Iterations -- control the computational cost of deriving the key. A higher iteration count makes brute-force attacks slower. Modern recommendations suggest at least 100,000 iterations for PBKDF2 with HMAC-SHA-256, though higher values (600,000+) provide better protection against GPU-accelerated attacks.

While newer key derivation functions like Argon2 offer better resistance to GPU and ASIC attacks by requiring large amounts of memory, PBKDF2 remains widely supported and is available natively in the Web Crypto API, making it the practical choice for browser-based tools.

The Link-Based Sharing Model

One of the most elegant aspects of browser-based encrypted notes is how the decryption key is delivered to the recipient. The key is placed in the URL fragment -- the portion of the URL after the # character.

According to the HTTP specification (RFC 3986), the fragment identifier is processed entirely by the client and is never sent to the server in HTTP requests. When you visit a URL like https://example.com/note/abc123#secretkey, your browser sends a request for /note/abc123 but the #secretkey part stays in the browser. This means:

  • The server never sees the decryption key in access logs.
  • The key is not visible to network intermediaries.
  • The key does not appear in server-side analytics or monitoring.

The recipient simply clicks the link, and their browser automatically has everything needed to fetch the ciphertext and decrypt it. No password entry is required unless an additional password layer has been configured.

Self-Destructing Notes

To further limit exposure, secure note tools often support self-destructing messages that are automatically deleted after being read once or after a set time period. The duckTyped Secure Note Sender supports this feature -- once the recipient opens and reads the note, the encrypted data is permanently removed from the server.

Self-destruction reduces the attack surface significantly. Even if the server's storage were compromised at a later date, previously read notes would no longer exist. Combined with the fact that the server never held the decryption key, this creates a strong defense-in-depth model: the ciphertext is transient, and the key was never stored server-side in the first place.

Zero-Knowledge Architecture

The term zero-knowledge in this context means that the server operator has zero knowledge of the content being stored. The server functions as a blind relay -- it stores and delivers encrypted blobs without any ability to inspect or understand them. Even if compelled by legal process or compromised by an attacker, the server cannot produce plaintext because it never possessed the decryption key.

This architecture requires trust only in the client-side code running in the browser. Users can verify the encryption implementation by inspecting the page source, since all cryptographic operations use the browser's built-in Web Crypto API rather than custom or obfuscated code.

When to Use Secure Notes

Encrypted notes are ideal for sharing sensitive information that should not persist in email inboxes, chat logs, or other message archives:

  • Passwords and credentials -- sharing login details with a colleague or client without leaving them in a Slack message or email thread.
  • API keys and tokens -- transmitting secrets that grant access to services or infrastructure.
  • Private personal information -- Social Security numbers, financial details, or medical information that should not be stored in plaintext anywhere.
  • One-time codes and PINs -- sharing temporary access codes that are only needed once.

Limitations

No security tool is perfect, and it is important to understand the boundaries of what encrypted notes can and cannot protect against:

  • Link interception -- if the link is shared over an insecure channel and intercepted, the attacker has both the ciphertext location and the key. Always share links over secure channels (encrypted messaging apps, HTTPS-protected email, etc.).
  • Endpoint compromise -- if the sender's or recipient's device is compromised by malware, encryption cannot help. A keylogger or screen capture tool can capture the plaintext before or after encryption.
  • No recipient authentication -- the tool does not verify the identity of the person opening the link. Anyone with the link can decrypt the note. For sensitive scenarios, consider adding a password known only to the intended recipient.
  • Browser trust -- the security model depends on the browser correctly implementing the Web Crypto API and the page code not being tampered with. Using a modern, updated browser mitigates this risk.

How to Use the Tool

The duckTyped Secure Note Sender makes it straightforward to share encrypted messages:

  1. Type or paste your secret into the message field. The content stays in your browser -- nothing is sent until you explicitly create the note.
  2. Configure options such as self-destruction after reading or an expiration time limit.
