HTTP Security Headers Explained

Understand what HTTP security headers are, how each one protects your website and users, learn recommended configurations, and use duckTyped's security headers checker to audit any site.

What Are Security Headers?

HTTP security headers are directives sent by a web server in its HTTP response that instruct the browser how to behave when handling the site's content. They act as a critical layer of defense against common web attacks including cross-site scripting (XSS), clickjacking, data injection, and protocol downgrade attacks.

Security headers are added to the HTTP response by the web server or application framework. They cost nothing to implement, require no changes to your application code, and provide immediate protection against entire classes of vulnerabilities. Despite this, many websites fail to deploy them correctly -- or at all.

Each header addresses a specific attack vector. Together, they form a defense-in-depth strategy that significantly reduces your site's attack surface.

Content-Security-Policy (CSP)

The Content-Security-Policy header is the most powerful and complex security header available. It defines an allowlist of content sources that the browser is permitted to load for a given page. Any resource (script, stylesheet, image, font, frame, etc.) that does not match the policy is blocked by the browser.

CSP is the primary defense against cross-site scripting (XSS) attacks. By restricting which scripts the browser is allowed to execute, CSP prevents injected malicious scripts from running -- even if an attacker successfully injects HTML into your page.

A basic CSP policy looks like this:

Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com; style-src 'self' 'unsafe-inline'; img-src *; frame-ancestors 'none'

Key directives include:

  • default-src -- fallback policy for all resource types not explicitly specified.
  • script-src -- controls which scripts can execute. Avoid 'unsafe-inline' and 'unsafe-eval' whenever possible, as they weaken CSP's XSS protection. Use nonces or hashes instead.
  • style-src -- controls which stylesheets can be applied.
  • img-src -- controls which image sources are allowed.
  • connect-src -- controls which URLs the page can contact via XHR, Fetch, or WebSocket.
  • frame-ancestors -- controls which sites can embed your page in an iframe (replaces X-Frame-Options).
  • report-uri / report-to -- specifies an endpoint where the browser sends violation reports, invaluable for monitoring and tuning your policy.

Recommended approach: Start with Content-Security-Policy-Report-Only to monitor violations without blocking content, then transition to an enforcing policy once you have resolved all legitimate violations.

Strict-Transport-Security (HSTS)

The Strict-Transport-Security header tells browsers to only communicate with your site over HTTPS, never HTTP. Once a browser receives this header, it will automatically upgrade all future HTTP requests to HTTPS for the specified duration -- even if the user types http:// in the address bar or clicks an HTTP link.

HSTS protects against protocol downgrade attacks and cookie hijacking. Without HSTS, an attacker on the same network (e.g., a public Wi-Fi hotspot) can intercept the initial HTTP request before it redirects to HTTPS and perform a man-in-the-middle attack.

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
  • max-age -- the duration in seconds that the browser should remember to enforce HTTPS. A value of 31536000 (one year) is standard.
  • includeSubDomains -- applies the HSTS policy to all subdomains as well.
  • preload -- signals that you want your domain added to the browser's built-in HSTS preload list, which enforces HTTPS from the very first visit (eliminating even the initial HTTP request). Submit your domain at hstspreload.org.

Recommended value: max-age=31536000; includeSubDomains; preload

X-Content-Type-Options

The X-Content-Type-Options header prevents browsers from MIME-type sniffing -- the practice of guessing a file's content type by examining its contents rather than trusting the declared Content-Type header. MIME sniffing can be exploited to execute malicious content disguised as a harmless file type.

For example, an attacker could upload a file with a .jpg extension that actually contains JavaScript. Without this header, some browsers might detect the JavaScript content and execute it.

X-Content-Type-Options: nosniff

Recommended value: nosniff -- there is only one valid value for this header, and it should always be set.

X-Frame-Options

The X-Frame-Options header controls whether your page can be embedded in an <iframe>, <frame>, <embed>, or <object> element on another site. This is the primary defense against clickjacking attacks, where an attacker overlays a transparent iframe of your site on top of a malicious page, tricking users into clicking buttons or links they cannot see.

X-Frame-Options: DENY

Valid values are:

  • DENY -- the page cannot be framed by any site, including itself.
  • SAMEORIGIN -- the page can only be framed by pages on the same origin.

Note that X-Frame-Options is being superseded by the CSP frame-ancestors directive, which offers more granular control. However, X-Frame-Options should still be set for compatibility with older browsers that do not support CSP.

Recommended value: DENY (or SAMEORIGIN if your site uses iframes internally).

Referrer-Policy

The Referrer-Policy header controls how much referrer information the browser includes when navigating away from your site. By default, browsers send the full URL of the referring page in the Referer header, which can leak sensitive information such as session tokens in URLs, internal page paths, or search queries.

Referrer-Policy: strict-origin-when-cross-origin

Common values include:

  • no-referrer -- never send the Referer header.
  • same-origin -- only send the referrer for same-origin requests.
  • strict-origin-when-cross-origin -- send the full URL for same-origin requests, only the origin for cross-origin requests over HTTPS, and nothing when downgrading from HTTPS to HTTP. This is the recommended default.
  • no-referrer-when-downgrade -- send the full URL except when navigating from HTTPS to HTTP.

Recommended value: strict-origin-when-cross-origin

Permissions-Policy

The Permissions-Policy header (formerly Feature-Policy) controls which browser features and APIs your page and its embedded iframes are allowed to use. This prevents malicious or compromised third-party scripts from accessing sensitive device capabilities.

Permissions-Policy: camera=(), microphone=(), geolocation=(), payment=(self)

Key features you can control:

  • camera -- access to the device camera.
  • microphone -- access to the device microphone.
  • geolocation -- access to the user's location.
  • payment -- use of the Payment Request API.
  • usb -- access to USB devices.
  • autoplay -- automatic playback of media.
  • fullscreen -- use of the Fullscreen API.

Setting a feature to () disables it entirely. Setting it to (self) allows it only for the site's own origin. You can also allowlist specific origins.

Recommended approach: Disable all features you do not use and only enable the ones your application requires.

X-XSS-Protection (Deprecated)

The X-XSS-Protection header was designed to activate the browser's built-in XSS filter, which attempted to detect and block reflected XSS attacks. However, this header is now deprecated and should be set to 0 to disable the filter entirely.

X-XSS-Protection: 0

The reason for disabling it is that the XSS auditor itself introduced security vulnerabilities in certain edge cases. Modern browsers (Chrome, Edge, Firefox) have removed the XSS auditor entirely. A properly configured Content-Security-Policy provides far superior XSS protection.

Recommended value: 0

CORS Headers Overview

Cross-Origin Resource Sharing (CORS) headers control which external origins are allowed to make requests to your API or access your resources from client-side JavaScript. While CORS headers are not strictly "security headers" in the same sense as CSP or HSTS, misconfiguring them can expose your API to unauthorized cross-origin access.

The key CORS headers are:

  • Access-Control-Allow-Origin -- specifies which origins can access the resource. Avoid using * for APIs that handle sensitive data or use credentials.
  • Access-Control-Allow-Methods -- specifies which HTTP methods (GET, POST, PUT, DELETE, etc.) are permitted for cross-origin requests.
  • Access-Control-Allow-Headers -- specifies which request headers are permitted in cross-origin requests.
  • Access-Control-Allow-Credentials -- indicates whether the browser should include cookies and authorization headers with cross-origin requests. When set to true, Access-Control-Allow-Origin must not be *.

A common security mistake is setting Access-Control-Allow-Origin: * on APIs that require authentication. Always restrict the allowed origin to the specific domains that need access.

How to Implement Security Headers

Security headers are configured at the web server or application level. Here are examples for common platforms:

Nginx

server {
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'" always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "DENY" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;
    add_header X-XSS-Protection "0" always;
}

Apache (.htaccess or httpd.conf)

Header always set Content-Security-Policy "default-src 'self'; script-src 'self'"
Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains; preload"
Header always set X-Content-Type-Options "nosniff"
Header always set X-Frame-Options "DENY"
Header always set Referrer-Policy "strict-origin-when-cross-origin"
Header always set Permissions-Policy "camera=(), microphone=(), geolocation=()"
Header always set X-XSS-Protection "0"

Express.js (Node.js)

const helmet = require('helmet');
app.use(helmet());

// Or configure individually:
app.use((req, res, next) => {
    res.setHeader('Content-Security-Policy', "default-src 'self'; script-src 'self'");
    res.setHeader('Strict-Transport-Security', 'max-age=31536000; includeSubDomains; preload');
    res.setHeader('X-Content-Type-Options', 'nosniff');
    res.setHeader('X-Frame-Options', 'DENY');
    res.setHeader('Referrer-Policy', 'strict-origin-when-cross-origin');
    res.setHeader('Permissions-Policy', 'camera=(), microphone=(), geolocation=()');
    res.setHeader('X-XSS-Protection', '0');
    next();
});
