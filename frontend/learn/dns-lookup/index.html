DNS Records Explained

Understand how the Domain Name System works, what each record type does, and how to use duckTyped's DNS checker to troubleshoot domain configuration issues.

What Is DNS?

The Domain Name System (DNS) is often described as the internet's phone book. Every time you type a domain name like example.com into your browser, DNS is responsible for translating that human-readable name into a machine-readable IP address such as 93.184.216.34. Without DNS, you would need to memorize numeric addresses for every website you visit.

DNS is a globally distributed, hierarchical database. When your browser needs to resolve a domain name, the request follows a specific path:

  1. Your device sends a query to a recursive resolver (typically operated by your ISP or a public service like 1.1.1.1 or 8.8.8.8).
  2. The recursive resolver contacts a root nameserver, which directs it to the appropriate top-level domain (TLD) server (e.g., the server responsible for .com).
  3. The TLD nameserver responds with the address of the domain's authoritative nameserver -- the server that holds the actual DNS records for the domain.
  4. The authoritative nameserver returns the requested record, and the resolver passes it back to your device.

This entire process typically completes in milliseconds, and results are cached at multiple levels to reduce latency for subsequent lookups.

DNS Record Types

DNS records are instructions stored on authoritative nameservers. Each record type serves a specific purpose. Below are the most common types you will encounter.

A Record

The A record is the most fundamental DNS record type. It maps a domain name to an IPv4 address (a 32-bit address like 93.184.216.34). When someone visits your website, the browser uses the A record to find the server's IP address. A domain can have multiple A records for load balancing, and these are sometimes called round-robin DNS.

AAAA Record

The AAAA record (pronounced "quad-A") is the IPv6 equivalent of the A record. It maps a domain name to an IPv6 address (a 128-bit address like 2606:2800:0220:0001:0248:1893:25c8:1946). As IPv4 address space has been exhausted, AAAA records are increasingly important for ensuring your domain is reachable over modern networks.

CNAME Record

A CNAME (Canonical Name) record creates an alias that points one domain name to another. For example, you might configure www.example.com as a CNAME pointing to example.com, so that both addresses serve the same content. When a resolver encounters a CNAME, it follows the alias and performs a second lookup on the target domain.

There is an important restriction: a CNAME record cannot coexist with other record types at the same name. This means you cannot place a CNAME at the zone apex (e.g., example.com) if you also need MX or TXT records there. Some DNS providers offer proprietary alternatives (like ALIAS or ANAME records) to work around this limitation.

MX Record

The MX (Mail Exchange) record specifies which mail servers are responsible for receiving email on behalf of a domain. Each MX record includes a priority value -- a lower number indicates higher priority. For example:

example.com.  MX  10  mail1.example.com.
example.com.  MX  20  mail2.example.com.

In this configuration, mail is delivered to mail1.example.com first. If it is unavailable, the sending server falls back to mail2.example.com. Correctly configured MX records are essential for email delivery.

TXT Record

The TXT record stores arbitrary text data associated with a domain. While originally intended for human-readable notes, TXT records now serve critical machine-readable functions:

  • SPF (Sender Policy Framework) -- specifies which mail servers are authorized to send email for the domain, helping prevent spoofing.
  • DKIM (DomainKeys Identified Mail) -- publishes public keys used to verify cryptographic signatures on outgoing email.
  • DMARC (Domain-based Message Authentication) -- defines policies for handling emails that fail SPF or DKIM checks.
  • Domain verification -- services like Google Workspace, Microsoft 365, and various SaaS platforms use TXT records to prove domain ownership.

NS Record

The NS (Nameserver) record delegates a DNS zone to specific authoritative nameservers. These records tell the rest of the internet where to find the definitive DNS data for a domain. For example, if your domain uses Cloudflare for DNS, your NS records might point to ada.ns.cloudflare.com and bob.ns.cloudflare.com. NS records are typically set at your domain registrar and must point to servers that actually host your zone file.

SOA Record

The SOA (Start of Authority) record contains administrative information about a DNS zone. Every zone has exactly one SOA record, and it includes:

  • Primary nameserver -- the authoritative server for the zone.
  • Admin email -- contact address for the zone administrator (the @ symbol is replaced with a dot).
  • Serial number -- a version number that increments when the zone is updated, used by secondary servers to detect changes.
  • Refresh, Retry, Expire timers -- control how often secondary nameservers check for updates and how long they serve cached data if the primary becomes unreachable.

CAA Record

The CAA (Certificate Authority Authorization) record specifies which Certificate Authorities (CAs) are permitted to issue SSL/TLS certificates for a domain. This is a security best practice that prevents unauthorized CAs from issuing certificates. For example:

example.com.  CAA  0 issue "letsencrypt.org"
example.com.  CAA  0 issuewild "letsencrypt.org"

The issue tag authorizes standard certificates, while issuewild controls wildcard certificate issuance. If no CAA record exists, any CA may issue a certificate for the domain.

How DNS Resolution Works

When you enter a URL in your browser, the following steps occur before the page loads:

  1. Browser cache -- the browser checks its own DNS cache for a recent result.
  2. Operating system cache -- if the browser cache misses, the OS-level DNS cache (also called the stub resolver) is consulted.
  3. Recursive resolver -- if no cached result exists locally, the query is sent to a recursive resolver, which performs the full lookup on your behalf.
  4. Root nameserver -- the resolver queries one of the 13 root server clusters, which responds with the address of the appropriate TLD nameserver.
  5. TLD nameserver -- the resolver queries the TLD server (e.g., the .com server), which responds with the address of the domain's authoritative nameserver.
  6. Authoritative nameserver -- the resolver queries the authoritative server, which returns the requested record (e.g., the A record with the IP address).
  7. Response -- the resolver caches the result and returns it to your device. The browser can now connect to the server.

Each DNS record has a TTL (Time to Live) value, measured in seconds, that determines how long resolvers and clients should cache the record before requesting a fresh copy. A TTL of 3600 means the record is cached for one hour. Lower TTLs allow faster propagation of changes but increase query volume to authoritative servers.

Common DNS Issues

DNS misconfigurations are among the most frequent causes of website and email outages. Here are the issues you are most likely to encounter.

Domain Not Resolving

If your domain returns an NXDOMAIN error or no results at all, the most likely cause is missing A or AAAA records. Verify that your DNS zone includes at least one A record pointing to your web server's IP address. Also check that your NS records are correctly configured at your registrar.

Email Not Working

Email delivery failures are commonly caused by missing or incorrect MX records. Ensure that your MX records point to valid, reachable mail servers and that the priority values are set correctly. If you use a hosted email service (Gmail, Outlook, etc.), follow the provider's specific MX configuration instructions.

SPF / DKIM / DMARC Failures

If your emails are landing in spam folders or being rejected, the problem often lies in TXT record misconfiguration. Common mistakes include having multiple SPF records (only one is allowed per domain), exceeding the SPF 10-lookup limit, or publishing DKIM keys with incorrect formatting. Use the DNS checker to verify that your TXT records are syntactically correct and contain the expected values.

SSL Certificate Issues

If a Certificate Authority refuses to issue or renew your SSL certificate, check your CAA records. A CAA record that does not include your CA will block certificate issuance. If you are unsure which CA you use, either add the appropriate CAA record or remove CAA records entirely (which allows any CA to issue).

Propagation Delays

After making DNS changes, it can take time for the updates to propagate globally. This delay is governed by the TTL of the previous record -- resolvers will continue serving the cached version until the TTL expires. If you anticipate making changes, lower the TTL in advance (e.g., to 300 seconds) to minimize propagation time. Full global propagation can take anywhere from minutes to 48 hours depending on caching behavior.
