Reverse DNS Explained

Understand how reverse DNS translates IP addresses back to hostnames, what PTR records are, how the in-addr.arpa and ip6.arpa zones work, why rDNS is critical for email deliverability and security verification, and how to use duckTyped's reverse DNS lookup tool.

What Is Reverse DNS?

Reverse DNS (rDNS) is the process of resolving an IP address back to a domain name -- the opposite of the standard forward DNS lookup. In forward DNS, you query a domain name like mail.example.com and receive an IP address like 93.184.216.34. In reverse DNS, you query the IP address 93.184.216.34 and receive the hostname mail.example.com.

While forward DNS is essential for finding where to connect, reverse DNS serves a different purpose: it provides a way to verify the identity of an IP address. When a server connects to your mail server, reverse DNS lets you check whether the connecting IP address has a legitimate hostname associated with it. This verification is a foundational element of email security, logging, and network administration.

PTR Records

Reverse DNS lookups are powered by PTR (Pointer) records. A PTR record maps an IP address to a hostname, functioning as the reverse counterpart of an A record (which maps a hostname to an IPv4 address) or an AAAA record (which maps a hostname to an IPv6 address).

Unlike A and AAAA records, which are managed by the domain owner, PTR records are managed by whoever controls the IP address block -- typically the ISP or hosting provider. This is an important distinction. If you want to set up reverse DNS for your server, you must configure the PTR record through your hosting provider or ISP, not through your domain registrar or DNS provider.

A PTR record for the IP address 93.184.216.34 would look like this in DNS zone format:

34.216.184.93.in-addr.arpa.  PTR  mail.example.com.

The in-addr.arpa and ip6.arpa Zones

Reverse DNS uses a clever mechanism to fit IP-to-hostname mappings into the existing DNS hierarchy. Rather than creating an entirely new system, reverse DNS repurposes the standard DNS tree using special domains.

IPv4: in-addr.arpa

For IPv4 addresses, reverse DNS uses the in-addr.arpa domain. The IP address octets are reversed and appended to in-addr.arpa. For example, to look up the reverse DNS for 93.184.216.34, you query:

34.216.184.93.in-addr.arpa

The octets are reversed because DNS is hierarchical from right to left (the most significant part is on the right), while IP addresses are hierarchical from left to right (the network portion comes first). Reversing the octets aligns the IP address hierarchy with the DNS hierarchy, allowing delegation to work correctly. The /24 network 93.184.216.0 corresponds to the DNS zone 216.184.93.in-addr.arpa, which can be delegated to the organization that controls that IP block.

IPv6: ip6.arpa

For IPv6 addresses, reverse DNS uses the ip6.arpa domain. The process is similar but more verbose -- each hexadecimal digit (nibble) of the full 128-bit address is reversed and separated by dots. For example, the IPv6 address 2001:0db8:85a3::8a2e:0370:7334 expands to:

4.3.3.7.0.7.3.0.e.2.a.8.0.0.0.0.0.0.0.0.3.a.5.8.8.b.d.0.1.0.0.2.ip6.arpa

This results in a 63-character domain name, which is unwieldy but works reliably within the existing DNS infrastructure. IPv6 reverse DNS is configured less frequently than IPv4, partly due to the vast address space making individual PTR records impractical for every address.

Forward DNS vs Reverse DNS

Forward and reverse DNS serve complementary purposes but operate independently:

  • Forward DNS (A/AAAA records): Maps a hostname to an IP address. Managed by the domain owner. Used to find where to connect.
  • Reverse DNS (PTR records): Maps an IP address to a hostname. Managed by the IP address owner (ISP/hosting provider). Used to verify identity.

Critically, forward and reverse DNS do not need to match. An IP address might have a PTR record pointing to server1.hosting.example, while the domain mywebsite.com has an A record pointing to that same IP. This is perfectly normal for shared hosting environments. However, for certain use cases -- particularly email -- matching forward and reverse DNS is important, as discussed below.

Why Reverse DNS Matters

Reverse DNS plays a vital role in several areas of internet infrastructure and security.

Email Deliverability

Reverse DNS is critically important for email. Most major email providers (Gmail, Microsoft 365, Yahoo, etc.) check the reverse DNS of connecting mail servers as part of their spam filtering. If your mail server's IP address has no PTR record, or if the PTR record does not match the server's hostname, your emails are significantly more likely to be rejected or classified as spam.

Many mail servers implement a strict policy: they will reject connections from IP addresses that have no PTR record at all. The logic is straightforward -- legitimate mail servers are almost always configured with proper reverse DNS, while spam botnets and compromised machines rarely are. If you run a mail server, configuring a correct PTR record is not optional; it is a baseline requirement for deliverability.

Logging and Auditing

Web servers, firewalls, and other network devices often use reverse DNS to convert IP addresses in log files to hostnames. This makes logs more readable and easier to analyze. Instead of seeing a connection from 203.0.113.42, an administrator sees a connection from crawler.searchengine.com, immediately identifying the traffic source. However, relying on rDNS for logging can introduce latency if lookups are slow, which is why many systems perform reverse DNS lookups asynchronously or cache results aggressively.

Security Verification

Reverse DNS is used as one component of security verification in various contexts. Network intrusion detection systems use rDNS to enrich alerts with hostname information. Access control lists sometimes reference hostnames rather than IP addresses, requiring reverse DNS lookups. SSH servers can be configured to verify that a connecting client's IP address has a PTR record matching its claimed identity.

Anti-Spam and Blacklisting

DNS-based blacklists (DNSBLs) like Spamhaus use the reverse DNS infrastructure pattern for their lookup mechanism. To check whether an IP address is blacklisted, you query the reversed IP against the blacklist's domain (e.g., 34.216.184.93.zen.spamhaus.org). This piggybacks on the same in-addr.arpa reversal concept, making blacklist checks extremely fast and efficient.

Forward-Confirmed Reverse DNS (FCrDNS)

Forward-Confirmed Reverse DNS is a verification technique that checks whether forward and reverse DNS are consistent. The process works as follows:

  1. Take the IP address and perform a reverse DNS lookup to get a hostname.
  2. Take that hostname and perform a forward DNS lookup to get an IP address.
  3. If the IP address from step 2 matches the original IP address from step 1, the reverse DNS is "forward-confirmed."

FCrDNS provides a stronger identity verification than reverse DNS alone, because it proves that the owner of the domain name has explicitly pointed it at that IP address. A malicious actor could potentially set a PTR record to claim any hostname, but they cannot control the forward DNS records for a domain they do not own. Many email servers and security systems use FCrDNS as a more reliable check than PTR records alone.

Configuring PTR Records

Setting up reverse DNS requires coordination with your IP address provider. The process varies depending on your hosting situation:

  • Cloud hosting (AWS, GCP, Azure) -- most cloud providers offer a self-service option to set PTR records for your allocated IP addresses through their control panel or API.
  • Dedicated servers -- your hosting provider typically provides a control panel option for setting PTR records, or you may need to submit a support request.
  • ISP-assigned IP addresses -- for static IPs from your ISP, you usually need to contact the ISP's support team to request a PTR record change.

When configuring a PTR record for a mail server, best practice is to set the PTR to the server's FQDN (Fully Qualified Domain Name), and ensure that FQDN has a matching A record pointing back to the same IP address. This establishes forward-confirmed reverse DNS.

Common Issues

Missing PTR Record

The most common reverse DNS issue is simply not having a PTR record at all. This is the default state for many IP addresses, especially those on residential ISP connections or newly provisioned cloud instances. For mail servers, this is a critical problem that will cause delivery failures.

Mismatched PTR and A Records

If the PTR record points to a hostname that does not resolve back to the same IP address (failing FCrDNS), some recipients will treat this as suspicious. Always verify that your PTR record and A record form a consistent pair.

Generic ISP Hostnames

Many ISPs automatically set PTR records for their IP addresses to generic hostnames like host-93-184-216-34.example-isp.net. While this is technically a valid PTR record, it does not help with email deliverability because it does not match your mail server's configured hostname. You need to replace this generic PTR with your server's actual FQDN.

Propagation Delays

After configuring a PTR record, it may take time to propagate. PTR records have TTL values just like any other DNS record, and resolvers will cache the previous value (or the absence of a value) until the TTL expires. Allow up to 24--48 hours for full propagation, though most changes take effect within a few hours.
