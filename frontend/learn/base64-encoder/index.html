Base64 Encoding Explained

Understand what Base64 encoding is, how the algorithm transforms binary data into ASCII text, how Unicode is handled, and when you should (and shouldn't) use it.

What Is Base64?

Base64 is a binary-to-text encoding scheme that represents arbitrary binary data using a set of 64 printable ASCII characters. It is formally defined in RFC 4648 and has been a cornerstone of data interchange on the web for decades.

The 64 characters used are: A-Z, a-z, 0-9, +, and /, with = serving as a padding character. Every combination of input bytes maps deterministically to an output string composed entirely of these safe, printable characters.

Base64 is not encryption. It provides no confidentiality whatsoever. Anyone with access to a Base64 string can decode it instantly. Its purpose is not to hide data but to ensure binary data can be transported safely through text-only channels -- systems that might corrupt or reject raw binary bytes, such as email protocols, JSON payloads, or URL parameters.

How Base64 Encoding Works

The Base64 algorithm converts binary data into text through a straightforward series of steps:

  1. Read the input as bytes. Each byte is 8 bits. The input "Hi" consists of two bytes: H = 72 (01001000) and i = 105 (01101001).
  2. Concatenate all bits. Join the bytes into a continuous bit stream. For "Hi": 01001000 01101001 = 16 bits total.
  3. Group into 6-bit chunks. Split the bit stream into groups of 6 bits. Since 16 is not evenly divisible by 6, pad the end with zeros to reach the next multiple of 6 (18 bits): 010010 | 000110 | 100100.
  4. Map each 6-bit value to the Base64 alphabet. Each 6-bit group represents a decimal value from 0 to 63. Look up the corresponding character: 18 = S, 6 = G, 36 = k.
  5. Apply padding. Base64 output must be a multiple of 4 characters. Since we produced 3 characters, append one = to reach 4.

The final result: "Hi" encodes to SGk=.

Input text:    H           i
ASCII value:   72          105
Binary:        01001000    01101001
                    |
Concatenated:  01001000 01101001 00  (padded to 18 bits)
                    |
6-bit groups:  010010  000110  100100
Decimal:       18      6       36
Base64 char:   S       G       k       =
                    |
Output:        SGk=

When the input length is a multiple of 3 bytes, no padding is needed. One byte of input requires two = pads; two bytes require one = pad; three bytes require none.

Base64url Variant

The standard + and / characters are problematic in URLs and filenames because they have special meaning in those contexts. The Base64url variant (also defined in RFC 4648) replaces them:

  • + becomes - (hyphen)
  • / becomes _ (underscore)

Base64url is used in JSON Web Tokens (JWTs), URL query parameters, and filename-safe contexts. Padding (=) is often omitted in Base64url since the decoder can infer it from the string length.

Handling Unicode Text

Base64 operates on raw bytes, not characters. For plain ASCII text (code points 0--127), each character maps directly to a single byte, so Base64 encoding is straightforward. However, Unicode text -- characters like accented letters, CJK scripts, or emoji -- requires an extra step.

Before Base64-encoding Unicode text, you must first convert the characters to their UTF-8 byte representation. UTF-8 encodes each character as one to four bytes. The Base64 algorithm then operates on those bytes.

In JavaScript, the built-in btoa() function only handles Latin-1 characters (code points 0--255) and will throw an error on anything beyond that range. The standard workaround is:

// Encode Unicode text to Base64
const encoded = btoa(unescape(encodeURIComponent(text)));

// Decode Base64 back to Unicode text
const decoded = decodeURIComponent(escape(atob(encoded)));

The duckTyped Base64 tool handles Unicode automatically -- you can paste text containing any characters, including emoji and non-Latin scripts, and it will encode and decode correctly without any extra steps.

Common Use Cases

Base64 encoding is used widely across web development and systems programming. Here are the most common applications:

  • Data URIs -- Embedding images, fonts, or other assets directly in HTML or CSS without a separate HTTP request: data:image/png;base64,iVBORw0KGgo.... This is common for small icons and inline SVGs.
  • Email Attachments -- The MIME standard uses Base64 to encode binary attachments (images, PDFs, archives) so they can travel through SMTP, which was originally designed for 7-bit ASCII text only.
  • API Payloads -- JSON does not support binary data natively. When you need to send a file, image, or binary blob in a JSON request body, Base64 encoding converts it to a text string that JSON can handle.
  • HTTP Basic Authentication -- The Authorization header for Basic Auth is constructed by Base64-encoding the username:password string: Authorization: Basic dXNlcjpwYXNz. Note that this provides no security on its own -- always use HTTPS.
  • Storing Binary in Text Formats -- Configuration files (YAML, XML, JSON), databases with text-only columns, environment variables, and other text-based storage can hold binary data when it is Base64-encoded.

Base64 Is Not Encryption

This point is worth emphasizing: Base64 provides zero security. It is a deterministic, publicly documented encoding with no key, no secret, and no randomness. Anyone can decode a Base64 string in milliseconds using freely available tools -- including the one on this site.

Never use Base64 to "hide" passwords, tokens, API keys, or any sensitive data. If you see a Base64-encoded string in a configuration file or URL, treat it as plaintext.

For actual confidentiality, use proper encryption algorithms such as AES-256-GCM with a strong key derivation function like PBKDF2 or Argon2. If you need to send a secret securely, consider using an end-to-end encrypted tool like the Secure Note Sender on duckTyped.

Size Overhead

Base64 encoding increases the size of data by approximately 33%. This happens because every 3 input bytes (24 bits) are represented as 4 output characters (4 bytes in ASCII). The ratio is exactly 4/3, or about 1.33x.

For small payloads like authentication tokens or short strings, this overhead is negligible. For larger data -- such as embedding a 500 KB image as a data URI -- the 33% increase becomes meaningful and can negatively impact page load times and bandwidth usage.

If you are transmitting Base64-encoded data over HTTP, gzip or Brotli compression applied at the transport layer can partially offset this overhead, since Base64 text compresses reasonably well. However, the compressed result will still typically be larger than the original binary data would have been, so for large files, serving them as actual binary (e.g., a separate image file) is almost always more efficient.
