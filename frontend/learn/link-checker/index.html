Broken Link Checking Explained

Understand what broken links are, how HTTP status codes signal errors, why dead links damage your site's SEO and user experience, and how to systematically find and fix them using duckTyped's Broken Link Checker.

What Are Broken Links?

A broken link (also called a dead link or link rot) is a hyperlink that no longer leads to its intended destination. When a user or search engine crawler follows a broken link, the server returns an error response instead of the expected content. The most familiar example is the 404 Not Found error, but broken links can manifest as several different HTTP status codes depending on the underlying cause.

Broken links are an inevitable part of the web. Pages get deleted, URLs get restructured, external sites shut down, and domains expire. A study by Harvard Law School found that roughly 50% of URLs cited in U.S. Supreme Court opinions no longer point to the original content. Research from the Internet Archive confirms that the average lifespan of a web page is approximately 100 days before it is modified or removed. This phenomenon -- known as link rot -- affects every website over time, making regular link checking an essential maintenance task.

HTTP Status Codes Explained

Every time a browser or crawler requests a URL, the server responds with an HTTP status code -- a three-digit number that indicates what happened. Understanding these codes is essential for diagnosing broken links.

1xx -- Informational

Status codes in the 100-199 range are informational responses. They indicate that the server has received the request and the client should continue. You rarely encounter these in the context of broken link checking, but 100 Continue and 101 Switching Protocols are the most common.

2xx -- Success

Codes in the 200-299 range indicate success. 200 OK is the standard response for a successful request -- this is what every working link should return. Other notable codes include 201 Created (used after successful POST requests) and 204 No Content (success, but no body to return).

3xx -- Redirection

The 300-399 range covers redirects. These are not errors per se, but they indicate the resource has moved. 301 Moved Permanently tells browsers and search engines to update their records to the new URL. 302 Found (temporary redirect) means the resource is temporarily at a different location. 307 Temporary Redirect and 308 Permanent Redirect are their modern equivalents that preserve the HTTP method. Redirect chains (multiple redirects in sequence) slow down page loading and can cause issues if they become circular.

4xx -- Client Errors

The 400-499 range indicates errors on the client side -- meaning the request itself is the problem. These are the codes most commonly associated with broken links:

  • 400 Bad Request -- the server cannot process the request due to malformed syntax, often caused by special characters in URLs or incorrect encoding.
  • 401 Unauthorized -- the resource requires authentication. The link itself may not be broken, but it is inaccessible without valid credentials.
  • 403 Forbidden -- the server understood the request but refuses to fulfill it. This often occurs when directory listing is disabled or when access is restricted by IP, geographic location, or permissions.
  • 404 Not Found -- the most iconic error code. The server cannot find the requested resource. This typically means the page was deleted, the URL was changed, or the link contains a typo.
  • 405 Method Not Allowed -- the HTTP method (GET, POST, etc.) is not supported for the requested resource. Some link checkers use HEAD requests for efficiency, which certain servers reject.
  • 410 Gone -- similar to 404, but explicitly states that the resource has been intentionally and permanently removed. Search engines treat 410 as a stronger signal to de-index the URL compared to 404.
  • 429 Too Many Requests -- rate limiting. The client has sent too many requests in a given time window. This is common when scanning a site too aggressively.

5xx -- Server Errors

The 500-599 range indicates server-side failures:

  • 500 Internal Server Error -- a generic catch-all for unexpected server failures. The link may work again once the server issue is resolved.
  • 502 Bad Gateway -- the server, acting as a gateway or proxy, received an invalid response from the upstream server. Common with reverse proxies and load balancers.
  • 503 Service Unavailable -- the server is temporarily unable to handle the request, typically due to maintenance or overload. Often accompanied by a Retry-After header.
  • 504 Gateway Timeout -- the gateway or proxy did not receive a timely response from the upstream server.

Why Broken Links Matter

Broken links are more than a cosmetic problem. They have tangible consequences for user experience, search engine optimization, and site credibility.

User Experience

Encountering a dead link is frustrating for users. Instead of finding the information they expected, they hit a dead end. Studies show that users who encounter errors are significantly more likely to leave a site entirely rather than navigate back and try a different path. Every broken link is a potential lost visitor, lost customer, or lost subscriber.

SEO Impact

Search engines like Google use links as a fundamental ranking signal. When a crawler encounters broken links on your site, it sends negative quality signals. While Google has stated that a few 404 errors will not directly harm your rankings, a pattern of broken links suggests poor site maintenance. More importantly, broken internal links waste crawl budget -- the number of pages a search engine will crawl on your site in a given time window. Every request spent on a dead URL is a request not spent discovering or re-indexing your actual content.

Link Equity Loss

When external sites link to a page that no longer exists, the link equity (sometimes called "link juice") that would have boosted your page's authority is lost. If a high-authority site links to your deleted page, that valuable SEO signal disappears unless you set up a proper redirect to a relevant replacement page.

Types of Broken Links

Broken links can be categorized by their location and function:

  • Internal broken links -- links within your own site that point to pages, resources, or routes that no longer exist. These are entirely within your control to fix.
  • External broken links -- links on your site that point to third-party websites that have gone offline, changed their URL structure, or removed content. These require replacing the link or removing the reference.
  • Broken image links -- image src attributes that point to files that have been moved, deleted, or renamed. These result in broken image icons on the page.
  • Broken anchor links -- links that point to a specific section of a page (using #fragment identifiers) where the target anchor no longer exists. The page loads, but the browser cannot scroll to the intended section.
  • Broken resource links -- references to CSS files, JavaScript files, fonts, or other assets that are missing. These can cause layout issues, broken functionality, or degraded performance.

Common Causes of Broken Links

Understanding why links break helps you prevent them in the first place:

  • Deleted or moved pages -- content is removed or reorganized without setting up redirects from the old URLs.
  • URL restructuring -- site redesigns or CMS migrations often change URL patterns (e.g., from /blog/2024/post-title to /articles/post-title).
  • Expired domains -- external sites you linked to may have let their domain registration lapse.
  • Typos in URLs -- manual entry of links during content creation introduces misspellings or incorrect paths.
  • Platform changes -- switching from one CMS or hosting platform to another without mapping old URLs to new ones.
  • Dynamic content removal -- e-commerce products that go out of stock, job listings that expire, or event pages that are taken down.
  • Case sensitivity -- some servers treat /Page and /page as different URLs. Moving between servers with different case sensitivity rules can break links.

How to Fix Broken Links

Once you have identified broken links, there are several strategies to address them:

Set Up 301 Redirects

The best solution for moved content is a 301 permanent redirect from the old URL to the new one. This preserves link equity, ensures users reach the correct page, and tells search engines to update their index. Implement redirects at the server level (in .htaccess, Nginx configuration, or your application's routing layer) for the fastest response.

Update Internal References

For internal broken links, update the href or src attributes throughout your site to point to the correct URLs. This is more efficient than relying on redirects for links you control directly.

Replace External Links

If an external resource has moved, update your link to the new URL. If the resource no longer exists anywhere, either find an alternative resource, link to an archived version on the Wayback Machine, or remove the link entirely.

Create a Custom 404 Page

A well-designed custom 404 page mitigates the damage of any remaining broken links. Include a search bar, links to popular pages, and clear navigation so users can find what they were looking for. Avoid generic "page not found" messages with no way forward.

Use Canonical URLs Consistently

Establish a canonical URL format for your site (with or without trailing slashes, with or without www) and stick to it. Inconsistent URL formats are a common source of links that work sometimes but not others.
