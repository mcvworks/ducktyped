Traceroute Explained

Understand how traceroute maps the network path between your machine and a destination, how TTL (Time to Live) drives the discovery process, how to read and interpret traceroute output, and how to use duckTyped's traceroute tool for network diagnostics.

What Is Traceroute?

Traceroute is a network diagnostic tool that reveals the path packets take from your computer to a destination host. While ping tells you whether a host is reachable and how long the round trip takes, traceroute shows every intermediate router (or "hop") along the way, along with the latency at each step. This makes traceroute invaluable for diagnosing where in the network a problem is occurring.

When you experience slow connections or intermittent failures to a remote server, the issue might not be with the destination server itself -- it could be with any router or link along the path. Traceroute lets you pinpoint exactly where packets are being delayed, dropped, or rerouted. The tool was originally written by Van Jacobson in 1987 and has been a staple of network troubleshooting ever since.

How TTL (Time to Live) Works

Traceroute exploits a fundamental feature of IP packets: the Time to Live (TTL) field. Despite its name, TTL does not represent a time value -- it is a hop counter. Every IP packet includes a TTL value, and each router that forwards the packet decrements the TTL by one. When the TTL reaches zero, the router discards the packet and sends an ICMP Time Exceeded message back to the sender.

Traceroute uses this behavior systematically:

  1. It sends a packet with TTL = 1. The first router decrements the TTL to zero and returns an ICMP Time Exceeded message. The traceroute program records the router's IP address and the round-trip time.
  2. It sends a packet with TTL = 2. The first router decrements to 1 and forwards it. The second router decrements to zero and sends back an ICMP Time Exceeded message.
  3. This process repeats, incrementing the TTL by one each time, until the packet reaches the destination host. The destination responds differently (with an ICMP Port Unreachable or Echo Reply, depending on the implementation), signaling that the trace is complete.

At each hop, traceroute typically sends three probes to capture multiple latency measurements, providing a more reliable picture of performance at each router.

ICMP vs UDP vs TCP Traceroute

Different traceroute implementations use different protocols for their probe packets. The choice of protocol affects which routers respond and what firewalls allow through.

ICMP Traceroute

The Windows tracert command uses ICMP Echo Request packets (the same packets used by ping). Each probe is an ICMP ping with an incrementing TTL. This is the simplest approach, but many firewalls and routers are configured to drop or deprioritize ICMP traffic, which can cause gaps in the traceroute output.

UDP Traceroute

The traditional Unix traceroute command sends UDP packets to high-numbered ports (typically starting at 33434). When the packet reaches the destination, the target host responds with an ICMP Port Unreachable message (since nothing is listening on that port), signaling the end of the trace. UDP probes may pass through firewalls that block ICMP, but they can also be blocked by firewalls that restrict outbound UDP.

TCP Traceroute

TCP traceroute (often implemented by tools like tcptraceroute or traceroute -T) sends TCP SYN packets to a specific port, commonly port 80 or 443. Since firewalls almost always allow traffic to web ports, TCP traceroute is the most likely to produce complete results through firewalls and NAT devices. This makes it particularly useful for tracing paths to web servers.

Reading Traceroute Output

A typical traceroute output consists of numbered lines, one per hop. Each line contains several pieces of information:

 1  router.local (192.168.1.1)  1.234 ms  0.987 ms  1.102 ms
 2  10.0.0.1 (10.0.0.1)  8.456 ms  8.234 ms  8.567 ms
 3  isp-core.example.net (203.0.113.1)  12.345 ms  11.987 ms  12.102 ms
 4  * * *
 5  cdn-edge.example.com (198.51.100.1)  15.678 ms  15.456 ms  15.789 ms
  • Hop number -- the sequential position of the router in the path (1 is the first router after your machine).
  • Hostname -- the reverse DNS name of the router's interface, if available. This often reveals the ISP or network operator.
  • IP address -- the IP address of the router's interface that processed the packet.
  • Latency measurements -- the round-trip time for each of the three probes, measured in milliseconds. These represent the time from your machine to that specific hop and back.

Understanding Timeouts and Asterisks

The * symbol in traceroute output indicates a timeout -- the probe packet was sent, but no response was received within the timeout period. This does not necessarily mean the hop is unreachable or that traffic is being dropped. Common reasons for timeouts include:

  • ICMP rate limiting -- many routers limit the rate at which they generate ICMP responses to prevent resource exhaustion. Your probe may have been deprioritized.
  • Firewall filtering -- the router or a firewall in front of it is configured to not respond to traceroute probes.
  • Router policy -- some network operators deliberately configure their routers to not participate in traceroute to obscure their network topology.

A single line of * * * is common and usually not cause for concern. However, if all remaining hops show timeouts and the destination is never reached, this indicates either the destination is truly unreachable or a firewall is blocking the specific type of probe being used.

Identifying Bottlenecks

The primary purpose of traceroute is to identify where in the network path performance degrades. Look for these patterns:

Sudden Latency Increase

If latency jumps significantly at a particular hop and remains high for all subsequent hops, the link between the previous hop and the high-latency hop is likely the bottleneck. For example, if hops 1--4 show 10 ms latency and hops 5--10 show 150 ms, the link between hops 4 and 5 is adding approximately 140 ms of delay. This often occurs at international submarine cable links or congested peering points.

High Latency at a Single Hop

If one hop shows high latency but subsequent hops return to normal levels, the elevated latency is likely caused by ICMP rate limiting on that specific router rather than actual congestion. The router deprioritizes generating ICMP responses but forwards your actual data traffic at full speed. Do not misinterpret this as a bottleneck.

Packet Loss at Intermediate Hops

If some probes at a hop return results while others time out (e.g., 12.3 ms * 11.8 ms), this suggests intermittent packet loss or ICMP rate limiting. If packet loss at a hop correlates with loss at the final destination, the intermediate hop may be the source of the problem.

Asymmetric Routing

An important caveat when interpreting traceroute results is that internet routing is often asymmetric. The path your packets take to reach a destination may be completely different from the path the response packets take to return. Traceroute only reveals the forward path -- you cannot assume the return path mirrors it.

This means that high latency at a particular hop could actually be caused by a slow return path rather than a slow forward path. The round-trip time reported by traceroute includes both directions, but you can only see the routers on the outbound leg. For this reason, running a traceroute from both ends (your machine to the server, and the server back to you) provides a more complete picture.

Common Patterns

ISP Handoff

Traceroute output typically shows your traffic passing through your local ISP's network, then transitioning to one or more transit providers, and finally arriving at the destination's network. The handoff points between networks (called peering points or internet exchange points) are often where latency increases, especially if the peering link is congested.

CDN Termination

When tracing to a website that uses a CDN (Content Delivery Network), the traceroute typically terminates at a CDN edge node close to your location rather than traveling to the origin server. This results in a short traceroute with low latency -- which is exactly the CDN's purpose. If you see a traceroute to a major website completing in just a few hops with very low latency, the CDN is doing its job.

Traceroute vs Ping

Ping sends ICMP Echo Request packets to a destination and measures the round-trip time, providing a simple connectivity test and overall latency measurement. Traceroute provides a much richer picture by revealing every hop along the path and the latency at each one. Use ping to quickly check if a host is reachable; use traceroute when you need to understand where in the network path a problem is occurring.
