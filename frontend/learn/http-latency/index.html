HTTP Latency Explained

Understand what HTTP latency is, how each phase of a web request contributes to total response time, what HTTP status codes mean, and how to use duckTyped's HTTP latency checker to measure and diagnose performance issues.

What Is HTTP Latency?

HTTP latency is the total time elapsed between a client sending an HTTP request and receiving the complete response from the server. It is commonly measured in milliseconds (ms) and is one of the most important metrics for web application performance. High latency leads to slow page loads, poor user experience, and lower search engine rankings.

Latency is distinct from bandwidth, which measures the volume of data that can be transferred per unit of time. A connection can have high bandwidth (capable of transferring large files quickly) but still suffer from high latency (each individual request takes a long time to start). Think of bandwidth as the width of a highway and latency as the time it takes to drive from one end to the other -- both affect your overall experience, but in different ways.

Components of HTTP Latency

A single HTTP request involves multiple sequential phases, each contributing to the total response time. Understanding these components is essential for diagnosing performance bottlenecks.

DNS Resolution

Before the browser can connect to a server, it must resolve the domain name to an IP address. This DNS lookup involves querying recursive resolvers and potentially authoritative nameservers. A cached DNS result may resolve in under 1 ms, while an uncached lookup can take 20--100 ms or more. DNS resolution happens once per domain and is then cached according to the record's TTL.

TCP Handshake

Once the IP address is known, the client initiates a TCP three-way handshake with the server. The client sends a SYN packet, the server responds with SYN-ACK, and the client sends a final ACK. This process typically takes one round-trip time (RTT) -- the time for a packet to travel from client to server and back. For a server 100 ms away, the TCP handshake alone adds approximately 100 ms of latency.

TLS Negotiation

For HTTPS connections (which account for the vast majority of modern web traffic), a TLS handshake follows the TCP handshake. During this phase, the client and server negotiate encryption parameters, exchange certificates, and establish a shared secret key. A TLS 1.2 handshake requires two additional round trips, while TLS 1.3 reduces this to a single round trip. On subsequent connections, TLS session resumption can further reduce or eliminate this overhead.

Time to First Byte (TTFB)

TTFB is the time from when the client sends the HTTP request to when it receives the first byte of the response. This metric reflects server-side processing time -- how long the server takes to receive the request, execute any application logic, query databases, generate the response, and begin sending it back. TTFB is often the most revealing metric for backend performance. A high TTFB indicates that the server is struggling to process requests, which could be caused by slow database queries, unoptimized application code, or resource contention.

Content Transfer

After the first byte arrives, the remaining response data must be transferred. The content transfer time depends on the response size and the available bandwidth. A small API response of a few kilobytes transfers almost instantly, while a large page with embedded resources may take significantly longer. Compression (gzip or Brotli) dramatically reduces transfer sizes for text-based content like HTML, CSS, and JavaScript.

HTTP Status Codes

When measuring latency, the HTTP status code in the response tells you whether the request succeeded and, if not, what went wrong. Status codes are grouped into five classes:

  • 1xx (Informational) -- the server has received the request and is continuing to process it. The most common is 100 Continue, which tells the client to proceed with sending the request body.
  • 2xx (Success) -- the request was successfully received, understood, and accepted. 200 OK is the standard success response. 201 Created indicates a new resource was created. 204 No Content means success with no response body.
  • 3xx (Redirection) -- the client must take additional action to complete the request. 301 Moved Permanently and 302 Found redirect to a different URL. 304 Not Modified indicates the cached version is still valid. Each redirect adds an additional round trip, increasing total latency.
  • 4xx (Client Error) -- the request contains an error on the client side. 400 Bad Request means malformed syntax. 401 Unauthorized requires authentication. 403 Forbidden means the server refuses to fulfill the request. 404 Not Found means the resource does not exist.
  • 5xx (Server Error) -- the server failed to fulfill a valid request. 500 Internal Server Error is a generic server failure. 502 Bad Gateway means an upstream server returned an invalid response. 503 Service Unavailable means the server is temporarily overloaded or under maintenance. 504 Gateway Timeout means an upstream server did not respond in time.

Factors Affecting Latency

Many variables influence the latency of an HTTP request. Understanding these factors helps you identify optimization opportunities.

Geographic Distance

The physical distance between client and server is a fundamental constraint. Data travels through fiber optic cables at roughly two-thirds the speed of light. A request from New York to a server in London (approximately 5,500 km) has a minimum theoretical round-trip time of about 55 ms, though real-world routing adds overhead. This is why geographic proximity to your users matters.

Server Processing Time

The time the server spends processing a request -- executing application code, querying databases, rendering templates -- directly impacts TTFB. Inefficient queries, memory pressure, or CPU contention can dramatically increase processing time, especially under heavy load.

Content Delivery Networks (CDNs)

A CDN distributes cached copies of your content across a global network of edge servers. When a user requests a resource, the CDN serves it from the nearest edge location rather than the origin server. This dramatically reduces latency for static assets (images, CSS, JavaScript) and can also cache dynamic content in some configurations. Major CDN providers include Cloudflare, AWS CloudFront, Fastly, and Akamai.

Caching

Caching at multiple levels reduces latency by serving previously computed results:

  • Browser cache -- stores resources locally based on Cache-Control and Expires headers. A cached resource has zero network latency.
  • CDN cache -- stores resources at edge locations, reducing the need to contact the origin server.
  • Server-side cache -- application-level caches (Redis, Memcached) store computed results to avoid repeated database queries or expensive calculations.

Connection Reuse

HTTP/1.1 introduced keep-alive connections, which allow multiple requests to reuse the same TCP connection, avoiding repeated handshakes. HTTP/2 goes further with multiplexing, allowing multiple requests and responses to be interleaved over a single connection. HTTP/3 uses QUIC (built on UDP) to eliminate head-of-line blocking and reduce connection establishment time to a single round trip.

Tips for Reducing Latency

Based on the factors above, here are practical strategies for reducing HTTP latency:

  1. Use a CDN to serve content from locations close to your users.
  2. Enable compression (gzip or Brotli) for all text-based responses.
  3. Optimize server-side code -- profile database queries, add caching layers, and reduce unnecessary computation.
  4. Minimize redirects -- each redirect adds a full round trip of latency.
  5. Upgrade to TLS 1.3 to reduce handshake overhead from two round trips to one.
  6. Set appropriate cache headers so browsers and CDNs can serve cached responses.
  7. Adopt HTTP/2 or HTTP/3 to benefit from multiplexing and reduced connection overhead.
  8. Reduce response size -- optimize images, minify CSS and JavaScript, and eliminate unnecessary data from API responses.
