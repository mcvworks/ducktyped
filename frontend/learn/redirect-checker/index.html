HTTP Redirects Explained

Understand how HTTP redirects work, the differences between redirect status codes, how redirect chains affect performance and SEO, and how to trace and diagnose redirect behavior using duckTyped's Redirect Checker.

What Are HTTP Redirects?

An HTTP redirect is a server response that instructs the client (typically a web browser) to request a different URL instead of the one originally requested. When a server returns a redirect status code (in the 3xx range), it includes a Location header specifying the new URL. The browser then automatically follows the redirect and requests the new URL, usually without any visible interruption to the user.

Redirects are a fundamental mechanism of the web. They enable websites to move content to new URLs without breaking existing links, enforce preferred URL formats (such as HTTPS or a canonical domain), and guide users through authentication and authorization flows. Nearly every website uses redirects in some form -- from simple HTTP-to-HTTPS upgrades to complex multi-step redirect chains in advertising and analytics tracking.

While redirects are essential, they must be implemented correctly. Misconfigured redirects can cause redirect loops (where the browser bounces endlessly between URLs), excessive redirect chains (where multiple hops slow down page loads), and SEO problems (where search engines lose track of where content actually lives).

Redirect Status Codes

HTTP defines several redirect status codes, each with different semantics that affect how browsers, search engines, and caching systems treat the redirect. Understanding these distinctions is essential for choosing the right redirect type.

301 -- Moved Permanently

The 301 redirect is the most commonly used redirect for permanent URL changes. It tells the client that the resource has been permanently moved to a new URL and that all future requests should use the new location. Search engines treat a 301 as a strong signal to transfer ranking authority (often called "link equity" or "link juice") from the old URL to the new one. Browsers cache 301 redirects aggressively, meaning subsequent visits to the old URL go directly to the new one without contacting the server.

Use 301 when: you have permanently moved a page to a new URL, changed your domain name, or restructured your site's URL hierarchy.

302 -- Found (Temporary Redirect)

The 302 redirect indicates that the resource is temporarily located at a different URL and the client should continue using the original URL for future requests. Despite its name ("Found"), this code has a complicated history. In HTTP/1.0, it was defined as "Moved Temporarily," but early browser implementations incorrectly changed POST requests to GET when following 302 redirects -- a behavior that became so widespread it was effectively standardized.

Use 302 when: a page is temporarily unavailable, during A/B testing, or when redirecting users based on geolocation or device type while preserving the original URL for later use.

303 -- See Other

The 303 redirect was introduced in HTTP/1.1 specifically to address the ambiguity of 302. It explicitly tells the client to retrieve the target resource using a GET request, regardless of the original request method. This is the correct status code to use after a POST request to redirect the user to a confirmation or result page -- a pattern known as the Post/Redirect/Get (PRG) pattern, which prevents duplicate form submissions when the user refreshes the page.

Use 303 when: redirecting after a form submission to prevent duplicate submissions on page refresh.

307 -- Temporary Redirect

The 307 redirect is the HTTP/1.1 replacement for 302 that explicitly preserves the request method. If the original request was a POST, the redirected request must also be a POST with the same body. This makes 307 the correct choice when temporarily redirecting an API endpoint or form action where maintaining the HTTP method is critical.

Use 307 when: you need a temporary redirect that strictly preserves the original request method (GET stays GET, POST stays POST).

308 -- Permanent Redirect

The 308 redirect is the permanent counterpart of 307. Like 301, it indicates the resource has permanently moved, but like 307, it strictly preserves the request method. A POST to a URL returning 308 must be retried as a POST to the new location. This code was defined in RFC 7538 to fill the gap left by 301's method-changing behavior.

Use 308 when: you need a permanent redirect that strictly preserves the original request method, particularly for API endpoints.

How Browsers Follow Redirects

When a browser receives a redirect response, it extracts the URL from the Location header and issues a new request to that URL. This process happens automatically and is usually invisible to the user -- the address bar updates to show the final URL, and the page loads as if the user had navigated there directly.

For each redirect hop, the browser performs a complete HTTP request-response cycle, including DNS resolution, TCP connection, TLS handshake (for HTTPS), and the actual data transfer. This means that each redirect adds latency to the page load. On mobile networks with high round-trip times, even a single unnecessary redirect can add hundreds of milliseconds to the loading time.

Browsers impose a maximum redirect limit to prevent infinite loops. Most browsers follow a maximum of 20 redirects before giving up and displaying an error ("too many redirects" or "ERR_TOO_MANY_REDIRECTS"). The HTTP specification does not mandate a specific limit but recommends that clients detect loops. In practice, the curl command-line tool defaults to a maximum of 50 redirects (configurable with the --max-redirs flag).

Redirect Chains

A redirect chain occurs when a URL redirects to another URL, which in turn redirects to yet another URL, and so on until a final destination is reached. For example:

http://example.com
  --> 301 --> https://example.com
    --> 301 --> https://www.example.com
      --> 301 --> https://www.example.com/home/

In this chain, the browser makes four requests to reach the final page. Each hop adds latency and consumes server resources. While a two-hop chain is sometimes unavoidable (such as HTTP to HTTPS plus www normalization), chains of three or more hops indicate an optimization opportunity.

To minimize chains, redirect directly to the final destination wherever possible. Instead of redirecting http://example.com to https://example.com and then to https://www.example.com, configure the first redirect to go straight to https://www.example.com.

Redirect Loops

A redirect loop occurs when URL A redirects to URL B, and URL B redirects back to URL A (or through a longer circular chain). The browser bounces endlessly between URLs until it hits the maximum redirect limit and displays an error.

Common causes of redirect loops include:

  • Misconfigured HTTP-to-HTTPS redirects -- if a load balancer or CDN terminates SSL and forwards requests to the origin server as HTTP, the origin server may redirect back to HTTPS, which the CDN again forwards as HTTP, creating an infinite loop.
  • Conflicting redirect rules -- one rule redirects www to non-www, while another redirects non-www to www.
  • CMS or plugin conflicts -- WordPress and other CMS platforms sometimes have redirect settings that conflict with server-level redirect rules.
  • Incorrect trailing slash handling -- one rule adds a trailing slash while another removes it, causing an endless loop.

Diagnosing redirect loops requires tracing the full redirect chain to identify the circular pattern. A redirect checker tool makes this straightforward by showing each hop in sequence.

SEO Implications of Redirects

Redirects have a significant impact on search engine optimization. Understanding how search engines treat different redirects is crucial for maintaining your site's ranking.

Link Equity Transfer

When external sites link to a URL that redirects, the ranking authority ("link equity" or "link juice") associated with those links needs to be transferred to the final destination. 301 redirects pass the vast majority of link equity to the target URL. Google has confirmed that 301 redirects transfer full PageRank. 302 redirects were historically treated differently -- search engines might keep the original URL indexed rather than the destination -- though Google now treats 302s and 301s similarly for link equity in most cases. Nonetheless, using the semantically correct redirect type is still best practice.

Crawl Budget

Crawl budget refers to the number of pages a search engine crawler will fetch from your site in a given time period. Each redirect in a chain consumes crawl budget because the crawler must follow each hop. Long redirect chains and redirect loops waste crawl budget, potentially causing important pages to be crawled less frequently. For large sites with thousands of pages, this can meaningfully impact indexing.

Canonical Signals

A 301 redirect is one of the strongest canonical signals you can send to search engines. It explicitly tells crawlers that the content has moved permanently and the new URL should be indexed. This is more definitive than a rel="canonical" tag, which is treated as a hint rather than a directive. When migrating content, a 301 redirect combined with a canonical tag on the destination page provides the clearest signal.

Common Redirect Patterns

Several redirect patterns are so common that they are considered standard web configuration:

HTTP to HTTPS

Redirecting all HTTP traffic to HTTPS is a fundamental security practice. This should be a 301 redirect because the change is permanent. The redirect is typically configured at the web server level (Apache, Nginx) or at the CDN/load balancer level. Once HTTPS is enforced, the site should also implement HSTS (HTTP Strict Transport Security) headers to instruct browsers to always use HTTPS, eliminating even the initial HTTP request on subsequent visits.

WWW to Non-WWW (or Vice Versa)

Most sites choose one canonical form -- either www.example.com or example.com -- and redirect the other to it using a 301. This prevents search engines from treating the www and non-www versions as separate sites with duplicate content. The choice between www and non-www is largely a preference, though the www version offers slightly more flexibility with DNS (CNAME records can be used at subdomains but not at the zone apex).

Trailing Slash Normalization

URLs with and without trailing slashes (e.g., /about vs /about/) are technically different URLs. Most sites pick one format and redirect the other to prevent duplicate content. Directory-style URLs typically use trailing slashes, while file-style URLs do not. Consistency is more important than which format you choose.

Meta Refresh vs JavaScript vs Server-Side Redirects

Not all redirects are implemented using HTTP status codes. Some are performed at the page level:

Server-Side Redirects (3xx Status Codes)

Server-side redirects are the recommended approach. The redirect happens before any page content is sent, making it fast and efficient. Search engines handle server-side redirects well, and browsers follow them instantly. Configure these in your web server (Apache .htaccess, Nginx config), application code, or CDN rules.

Meta Refresh

A <meta http-equiv="refresh" content="0;url=https://example.com"> tag in the HTML head instructs the browser to navigate to a new URL after a specified delay (in seconds). While functional, meta refreshes are slower than server-side redirects because the browser must download and begin parsing the HTML before encountering the redirect instruction. Search engines generally follow meta refreshes but may not pass link equity as effectively as 301 redirects. The content="0" value indicates an immediate redirect, but there is still more latency than a server-side redirect.

JavaScript Redirects

A JavaScript redirect like window.location.href = "https://example.com" executes a redirect from the client side. This is the slowest redirect method because the browser must download, parse, and execute the JavaScript before the redirect occurs. Search engines may not follow JavaScript redirects reliably, and no link equity transfer is guaranteed. JavaScript redirects should be used only as a last resort when server-side configuration is not possible.
